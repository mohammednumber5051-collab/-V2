### المشكلة #2: عدم توازن الدين والائتمان في القيود المحاسبية
**الملف:** `src/services/financialEngine.ts` (السطور 56-96)  
**الخطورة:** 🔴 **حرجة جداً**

#### الوصف:
في معاملات فاتورة المبيعات، يتم إنشاء معاملتان:

```typescript
// 1. معاملة الإثبات (debit الإيرادات)
transactions.push({
    type: 'قبض',
    debit: netAmount,    // الدين
    credit: 0,          // الائتمان
});

// 2. معاملة السداد (credit الصندوق)
if (paid > 0) {
    transactions.push({
        type: 'قبض',
        debit: 0,       // الدين
        credit: paid,   // الائتمان
    });
}
```

**المشكلة:** في نظام محاسبي سليم:
- **Debit (مدين)**: جانب اليسار (الأصول والتكاليف تزداد)
- **Credit (دائن)**: جانب اليمين (الخصوم والإيرادات تزداد)

الصيغة الصحيحة لفاتورة مبيعات:
```
من الصندوق (اليسار/Debit):    +المبلغ المدفوع
إلى الإيرادات (اليمين/Credit): +الإيرادات
```

#### الخطأ الحالي:
- معاملة 1: `debit: netAmount, credit: 0` (خطأ - يجب أن تكون معكوسة)
- معاملة 2: `debit: 0, credit: paid` (خطأ - يجب أن تكون معكوسة)

### المشكلة #3: حساب غير صحيح لأرصدة الذمم المدينة والدائنة
**الملف:** `src/lib/financialUtils.ts` (السطور 70-120)  
**الخطورة:** 🔴 **حرجة جداً**

#### الوصف:
دالة `calculateUnifiedPartnerBalances` تحتسب الأرصدة على النحو التالي:

```typescript
// السطور 119-125 (معالجة فاتورة)
const netAmount = (inv.total || 0) - (inv.discount || 0);
const sign = isReturn ? -1 : 1;
partnerBalances[inv.partnerId].total += (netAmount * sign);
partnerBalances[inv.partnerId].paid += ((inv.paid || 0) * sign);
```

**المشكلة في التعريف:**
- `total`: المبلغ الإجمالي للفاتورة ✓
- `paid`: المبلغ المدفوع ✓
- `remaining`: الفرق (total - paid) ✓

لكن **في الممارسة الحسابية**:
- عندما نضيف معاملة (قبض)، نطرح من `paid` الخاص بالعميل في التراساكشن
- لكن في الفاتورة، نضيف إلى `paid` مباشرة
- **النتيجة:** تكريار الخصم (double debit)!

## 📊 جدول التوازن (Balance Sheet Verification)

| البند | المعادلة الصحيحة | الحالة الحالية | الحالة |
|------|-----------------|----------------|--------|
| رصيد العميل | -المبلغ المقبوض | +المبلغ (معاكس!) | ❌ خطأ |
| رصيد المورد | +المبلغ المسدد | -المبلغ (معاكس!) | ❌ خطأ |
| رصيد الصندوق | +القبض, -الصرف | غير متسق | ❌ خطأ |
| الدين والائتمان | Debit = Credit | غير متساوي | ❌ خطأ |
| إجمالي الذمم المدينة | مجموع مديونيات العملاء | قد يكون مضاعف | ⚠️ تحذير |
| إجمالي الذمم الدائنة | مجموع مديونيات الموردين | قد يكون مضاعف | ⚠️ تحذير |

## 🔍 نقاط الضعف في الهيكل المعماري

### 1. **عدم وجود Normalization للبيانات**
- القيم مخزنة في أماكن متعددة (فاتورة + معاملة + رصيد شريك)
- عند التحديث، قد لا يتم تحديث جميع الأماكن
- **الحل:** استخدام "source of truth" واحد

### 2. **عدم وجود Transactional Consistency**
- عمليات متعددة قد تفشل في منتصفها
- لا يوجد rollback لجميع التحديثات
- **الحل:** استخدام Firestore Transactions بشكل صحيح

### 3. **عدم وجود Audit Trail محاسبي**
- لا يوجد سجل محاسبي كامل للمعاملات
- لا يمكن تتبع من قام بالتغيير متى
- **الحل:** تطبيق نظام Audit Logs شامل

### 4. **عدم تطبيق المبادئ المحاسبية**
- الدين والائتمان غير متوازنين
- لا يوجد اثبات القيد المزدوج (Double Entry)
- **الحل:** تطبيق نظام محاسبة بموجب نظام القيد المزدوج

### أمثلة عملية توضح المشاكل:

#### مثال 1: فاتورة مبيعات بـ 1000 ريال

**العملية:**
1. إنشاء فاتورة مبيعات للعميل "أحمد" بـ 1000 ريال
2. دفع 700 ريال الآن

**النتيجة المتوقعة:**
```
العميل أحمد:
  - إجمالي الفاتورة: 1000 ريال
  - المدفوع: 700 ريال
  - المتبقي: 300 ريال
  - رصيد العميل: +1000 (مدين بـ 1000 ريال للمتجر)
```

**النتيجة الحالية (خطأ):**
```
العميل أحمد:
  - إجمالي الفاتورة: 1000 ريال
  - المدفوع: 700 ريال
  - المتبقي: 300 ريال
  - رصيد العميل: -1000 (❌ معاكس!)
```

#### مثال 2: تحويل من صندوق لآخر

**العملية:**
1. تحويل 500 ريال من الصندوق A إلى الصندوق B

**النتيجة المتوقعة:**
```
الصندوق A: -500 ريال
الصندوق B: +500 ريال
المجموع: لا يتغير (0 صافي)

السجلات:
  - صرف من A: 500 ريال
  - قبض في B: 500 ريال
  - التوازن: صرف 500 = قبض 500 ✓
```

**النتيجة الحالية (خطأ):**
```
الصندوق A: -500 ريال ✓
الصندوق B: +500 ريال ✓
المجموع: صحيح ✓

السجلات:
  - صرف من A: 500 ريال ✓
  - قبض في B: لا يوجد! ❌
  - التوازن: صرف 500 ≠ قبض 0 ❌
```
